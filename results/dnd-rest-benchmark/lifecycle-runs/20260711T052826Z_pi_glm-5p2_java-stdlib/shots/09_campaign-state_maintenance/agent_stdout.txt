Here's a summary of what I added for **Stage 6: Campaign State APIs**:

### New Endpoints
| Method | Path | Status | Description |
|--------|------|--------|-------------|
| `POST` | `/v1/campaigns` | 201 | Create campaign (`id`, `name`, `dm`) |
| `POST` | `/v1/campaigns/{id}/characters` | 201 | Add character (`id`, `name`, `level`, `class`) |
| `POST` | `/v1/campaigns/{id}/events` | 201 | Add log event (`id`, `kind`, `summary`) → returns `{id, kind}` only |
| `GET` | `/v1/campaigns/{id}/state` | 200 | Read full campaign state with characters + `log_count` |

### Error handling
- **400** — malformed/missing fields (e.g., `level` as string, missing `summary`)
- **404** — unknown campaign (for character/event/state operations)
- **409** — duplicate campaign/character/event IDs

### Design decisions
- **In-memory `ConcurrentHashMap`** for campaigns (authoritative runtime store), with **write-through SQLite mirror** via the `sqlite3` CLI — same pattern as the existing `Store` class
- **`LinkedHashMap`** for characters/events within a campaign to preserve insertion order (matching spec output order)
- **`synchronized(camp)`** for character/event mutations — same concurrency pattern as `CombatSession`
- Named the character class **`GameCharacter`** (not `Character`) to avoid shadowing `java.lang.Character`, which the JSON parser relies on for `isWhitespace`/`isDigit` 🔍
- Used **`asInt`** (strict Long-only) for `level` — consistent with existing integer validation
- Added `CAMPAIGNS.clear()` to `storageReset` and new SQLite tables (`campaigns`, `characters`, `events`) to the schema
