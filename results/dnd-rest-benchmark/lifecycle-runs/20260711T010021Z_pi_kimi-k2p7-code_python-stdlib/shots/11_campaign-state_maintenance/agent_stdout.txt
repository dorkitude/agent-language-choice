I added the SQLite-backed Campaign State APIs while preserving all existing endpoints.

**Changes in `server.py`:**

- New schema tables: `campaigns`, `characters`, `events`
- New `Storage` methods:
  - `create_campaign`, `get_campaign`, `get_campaign_state`
  - `create_character`, `get_characters`
  - `create_event`, `get_event_count`
- New HTTP routes:
  - `POST /v1/campaigns` → creates a campaign
  - `POST /v1/campaigns/<id>/characters` → adds a character
  - `POST /v1/campaigns/<id>/events` → adds a session log event
  - `GET /v1/campaigns/<id>/state` → returns full campaign state with `characters` and `log_count`

Validation follows the existing conventions:
- 400 for malformed input
- 404 for unknown campaigns
- 409 for duplicate campaign/character/event IDs

`./run.sh` is already correct and executable — it launches the server in the foreground, and `server.py` reads `PORT` from the environment.

I verified the new endpoints and a few prior endpoints (`/v1/dice/stats`, `/v1/auth/register`, `/v1/storage/reset`) all work as expected.
