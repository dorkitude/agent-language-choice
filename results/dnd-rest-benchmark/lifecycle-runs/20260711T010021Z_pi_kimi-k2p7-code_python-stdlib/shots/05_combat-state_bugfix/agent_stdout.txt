Fixed, Chef! 🎲✨

**What was wrong:** The `advance` response was built by `conditions_response()`, which dropped any combatant whose condition list had become empty. When `blessed` expired on `fighter`, the `"fighter"` key vanished entirely, so the evaluator reported `missing JSON key "fighter"`.

**What changed in `server.py`:**
- Added a `condition_targets` set to each `CombatSession`.
- When a condition is added, the target name is recorded in that set.
- `conditions_response()` now returns an entry for every combatant that has **ever** had a condition, even if its current list is empty.

So after the condition expires, the response now correctly contains:

```json
"conditions": {"fighter": []}
```

I verified the full combat flow (create → add condition → advance through the order → condition expires) and all the prior core/character endpoints still behave as expected. `./run.sh` is ready and will start the server in the foreground on `127.0.0.1:$PORT`.
