I added the campaign-state APIs to the existing Go stdlib D&D REST server:

- **`POST /v1/campaigns`** — creates a campaign, returns `{id, name, dm}`
- **`POST /v1/campaigns/{id}/characters`** — adds a character, returns the full character
- **`POST /v1/campaigns/{id}/events`** — adds a log event, returns `{id, kind}`
- **`GET /v1/campaigns/{id}/state`** — returns campaign summary with sorted characters and `log_count`

I extended the existing JSON-backed store (`game.db`) with a `Campaigns` map, preserving the `"driver": "sqlite"` storage status already in use. All prior endpoints remain untouched.

Validation:
- Missing/empty fields → `400`
- Unknown campaign → `404`
- Duplicate campaign/character/event IDs → `409`

I verified the full happy path and error cases against a running server, plus persistence across restart, then stopped the server before finishing.

`./run.sh` is ready and unchanged — it builds and starts the server on `127.0.0.1:$PORT` in the foreground.
