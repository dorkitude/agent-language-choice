I added the three combat-state endpoints to `dndsite/urls.py` while preserving every existing endpoint from the core and character-rule suites:

- `POST /v1/combat/sessions` — creates a session with client-supplied id, computes initiative (`roll + dex`), sorts by score desc → dex desc → name asc, and starts at `round=1`, `turn_index=0`.
- `POST /v1/combat/sessions/{id}/conditions` — attaches a condition with positive `duration_rounds` to a named combatant.
- `POST /v1/combat/sessions/{id}/advance` — advances `turn_index`, wraps the round, and decrements/removes conditions on the newly active combatant at the start of its turn.

State lives in an in-memory `COMBAT_SESSIONS` dict. Unknown session IDs return `404`; malformed input returns `400`. Existing endpoints like `/v1/checks/ability`, `/v1/characters/derived-stats`, etc. still behave the same.

`./run.sh` is already in place and ready — it starts Django’s runserver on `127.0.0.1:$PORT` in the foreground. I did not start the server.
