## Summary

**The bug:** In `advanceTurn` (`src/server.ts`), when a combatant's conditions all expired, the code did two things that erased them from the response:
1. `session.conditions.delete(active.name)` вЂ” removed the combatant's entry entirely
2. The response builder skipped entries with `if (conds.length > 0)`

So when the fighter's `blessed` condition ticked `1 в†’ 0` on its second turn, the response became `conditions: {}`, but the evaluator's `combat-advance-expired-removed` test expects `conditions: {"fighter": []}` вЂ” the target's key must remain with an empty array to show "the expired condition was removed, and this target now has none."

**The fix:** Keep the combatant's entry in the conditions map with the filtered (possibly empty) list instead of deleting it, and include every map entry in the response:

```ts
const filtered = list.filter((c) => c.remaining_rounds > 0);
session.conditions.set(active.name, filtered);   // no more .delete()
...
for (const [name, conds] of session.conditions) {
  conditionsResp[name] = conds.map(...);          // no more length>0 guard
}
```

This matches the spec ("Remove a condition when its remaining duration reaches 0" вЂ” the *condition* is removed, not the combatant's slot) and is safe for all other tests, since the evaluator's `jsonContains` is a subset matcher that only checks for the *presence* of wanted keys.
