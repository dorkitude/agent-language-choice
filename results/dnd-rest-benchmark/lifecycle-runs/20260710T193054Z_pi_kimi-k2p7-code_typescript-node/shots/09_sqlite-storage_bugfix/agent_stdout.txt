Fixed it, Chef! 🍳✨

**The bug:** `getConditionsForSession` only built condition keys from rows still present in SQLite. When a condition expired and was deleted, the combatant's key disappeared from the `conditions` object entirely — so the evaluator saw `conditions: {}` instead of `conditions: {"fighter": []}`.

**The fix:** In `src/server.ts`, `getConditionsForSession` now pre-seeds the result with every combatant in the session (empty array), then attaches any remaining active conditions. This matches the prior in-memory stage behavior and keeps keys for combatants whose conditions have expired.

**Verified:**
- `npx tsc` compiles cleanly ✅
- Manual end-to-end run reproducing `round: 2, turn_index: 2, active: fighter` now returns `conditions: {"mage": [], "cleric": [], "fighter": []}` ✅

**`./run.sh` is ready:**
```bash
#!/usr/bin/env bash
set -euo pipefail
npx tsc && node dist/server.js
```

It starts the server on `127.0.0.1:$PORT` in the foreground. No extra frameworks added.
