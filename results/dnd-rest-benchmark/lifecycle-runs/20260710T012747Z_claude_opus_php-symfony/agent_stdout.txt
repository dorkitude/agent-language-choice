## What was broken 🐛

When the active combatant's last condition expired, the advance handler did `unset($session['conditions'][$activeName])`, deleting the whole key. The evaluator expects the combatant's key to *remain* with an empty list (`"fighter": []`) — a distinction between "never had conditions" (absent) and "had conditions, all expired" (present but empty).

## The fix ✨ (two minimal edits in `index.php`)

1. **`handle_combat_advance`** — stopped removing the key on expiry; now always writes the remaining list (even if `[]`).
2. **`conditions_view`** — switched from `!empty(...)` to `array_key_exists(...)`, so a combatant that has ever carried a condition stays in the response map even once its list empties out.

Because the evaluator's matcher does subset checking, combatants that never had conditions still don't appear — so the earlier passing cases (`combat-advance-condition-expires`, etc.) remain green. `run.sh` is unchanged and ready. 🚀
