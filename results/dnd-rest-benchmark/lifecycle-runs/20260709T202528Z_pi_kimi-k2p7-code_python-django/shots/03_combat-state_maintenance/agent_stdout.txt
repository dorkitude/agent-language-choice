I added the **Stateful Combat** stage to the Django project while preserving all existing endpoints.

### What changed
- **`dndsite/views.py`**:
  - Added in-memory `COMBAT_SESSIONS` storage with a thread-safe lock.
  - Added `create_combat_session`, `add_condition`, and `advance_turn` views.
  - Updated the `_require_post` decorator to pass through URL path parameters so the new parameterized routes work.
  - Restored the `@_require_post` decorator on `derived_stats` (it had been accidentally dropped during the helper refactor).

- **`dndsite/urls.py`**:
  - Wired up the three new routes:
    - `POST /v1/combat/sessions`
    - `POST /v1/combat/sessions/<session_id>/conditions`
    - `POST /v1/combat/sessions/<session_id>/advance`

- **`run.sh`**: unchanged — it already starts the server in the foreground on `127.0.0.1:$PORT`.

### Verified behavior
- Initiative is sorted by `score` desc, `dex` desc, `name` asc.
- New sessions start at `round=1`, `turn_index=0`.
- Duplicate session IDs return `400`.
- Unknown session IDs return `404`.
- Conditions attach to the target combatant and decrement only on the active combatant's turn.
- Conditions are removed when `remaining_rounds` reaches `0`.
- Turn wrap increments the round.
- All existing endpoints (health, dice, checks, encounters, initiative, characters) still pass.

I did not start the server — `./run.sh` is ready to go when the evaluator runs it.
