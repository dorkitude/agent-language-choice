## Summary

I inherited the existing Sinatra D&D REST API (Ruby 4.0.5 / Sinatra 4.2.1 / Rack 3.2.6 / Puma 8.0.2) and added the three **Stateful Combat** endpoints for this maintenance stage, preserving all prior core + character-rules behavior.

### What I added (`app.rb`)

An in-memory session store (`COMBAT_SESSIONS` hash + `COMBAT_LOCK` mutex for Puma thread-safety) and three routes:

- **`POST /v1/combat/sessions`** вЂ” validates `id` (non-empty string) and a non-empty `combatants` array (each with string `name`, int `dex`, int `roll`). Computes `score = roll + dex`, sorts by **score desc, dex desc, name asc**, stores `round=1`, `turn_index=0`, and returns `id`/`round`/`turn_index`/`active`/`order`.
- **`POST /v1/combat/sessions/:id/conditions`** вЂ” 404 on unknown session; validates `target` (string naming a real combatant), `condition` (arbitrary string), `duration_rounds` (positive int). Appends to the target's condition list and returns the full list.
- **`POST /v1/combat/sessions/:id/advance`** вЂ” 404 on unknown session; bumps `turn_index`, wraps to `0` and increments `round` at the end, then **decrements only the newly-active combatant's conditions**, removing any that hit `0`. Returns the full conditions map (only combatants with remaining conditions).
