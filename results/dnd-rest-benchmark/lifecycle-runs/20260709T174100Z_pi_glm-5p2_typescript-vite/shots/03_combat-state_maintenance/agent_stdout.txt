## Summary 🎲

Done, Chef! I added the **Stateful Combat** stage to the existing Vite dev-server middleware in `src/api.ts`, preserving all prior core + character-rule endpoints.

### What I added

**In-memory state** — a `Map<string, CombatSession>` keyed by client-supplied id, each holding the sorted initiative `order`, `round`, `turn_index`, and a per-combatant `conditions` map.

**Three new endpoints:**

| Endpoint | Behavior |
|---|---|
| `POST /v1/combat/sessions` | Validates `id` + non-empty `combatants`; computes `score = roll + dex`; sorts by **score desc → dex desc → name asc**; starts `round=1, turn_index=0`; returns `active` + `order`. |
| `POST /v1/combat/sessions/{id}/conditions` | 404 for unknown session; 400 for unknown target / non-string condition / non-positive-integer duration; appends condition; returns the target's full condition list. |
| `POST /v1/combat/sessions/{id}/advance` | 404 for unknown session; advances `turn_index` (wraps → `round++`); **decrements only the newly-active combatant's conditions** at the start of their turn; removes conditions at 0; returns full non-empty conditions map. |

**Router change** — kept the exact-match `POST_ROUTES` table (now including `/v1/combat/sessions`) and added a small `DYNAMIC_POST_ROUTES` regex list for the `{id}/conditions` and `{id}/advance` path-param routes. Unknown `/v1/*` routes still return 404; non-API paths still fall through to Vite.
