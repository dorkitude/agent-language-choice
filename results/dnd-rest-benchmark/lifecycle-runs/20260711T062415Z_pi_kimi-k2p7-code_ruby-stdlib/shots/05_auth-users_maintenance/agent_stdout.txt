Done, Chef! 🎲🔐

I added the `/v1/auth/register` and `/v1/auth/login` endpoints to `server.rb` while keeping every existing D&D endpoint intact.

What changed:
- `POST /v1/auth/register` — validates username (`2-32` chars, lowercase letters/digits/_/-), password (`≥8` chars), and role (`dm` or `player`), hashes the password with PBKDF2-SHA256 via `OpenSSL::KDF`, and returns `409` for duplicate usernames.
- `POST /v1/auth/login` — returns the deterministic token `session-<username>` on valid credentials, `401` otherwise.
- Password hashing lives in tiny helpers (`generate_salt`, `hash_password`, `verify_password`) so it can be swapped for a production hasher later.
- Added `401`/`409` to `STATUS_TEXT` and `USERS` in-memory store.

No Sinatra, Rails, Rack, or gems — just `json`, `socket`, and `openssl`.

`run.sh` is ready and foreground-friendly:

```bash
#!/usr/bin/env bash
set -euo pipefail
ruby server.rb
```

I did a quick smoke test: health, register, login, duplicate username, and bad password all behaved correctly. The server is **not** running now — it’s ready for you to launch.
