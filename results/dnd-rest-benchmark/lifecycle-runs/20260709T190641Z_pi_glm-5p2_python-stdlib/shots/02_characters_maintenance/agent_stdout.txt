## Summary

I added the **Character Rules** stage to the existing D&D REST API (Python 3.14 stdlib only вЂ” `http.server` + `json`), preserving all 8 core-suite endpoints.

### New endpoints (all `POST`, JSON in/out, `400` on invalid input)

| Route | Handler | Spec |
|---|---|---|
| `/v1/characters/ability-modifier` | `ability_modifier` | `score` int 1вЂ“30 в†’ `modifier = (score-10)//2` (Python `//` floors negative halves correctly: `9 в†’ -1`) |
| `/v1/characters/proficiency` | `proficiency` | `level` int 1вЂ“20 в†’ `2 + (level-1)//4` gives the 2/3/4/5/6 bands |
| `/v1/characters/derived-stats` | `derived_stats` | Combines both: 6 ability modifiers, proficiency, `hp_max = level*(6+con_mod)`, `armor_class = base + min(dex_mod, dex_cap) + shield_bonus` |

### Key design choices
- **Strict integer validation**: a `_is_int` helper accepts real JSON ints and explicitly rejects `bool` (Python's `bool` subclasses `int`). Out-of-range, missing, string, and float scores/levels all return `400`.
- **Floor correctness**: integer floor division `//` naturally floors negative halves, so `7 в†’ -2` and `1 в†’ -5`.
- **Shared formulas**: `ability_modifier_of` / `proficiency_bonus_of` are reused by both the standalone endpoints and `derived-stats`, guaranteeing identical results.
- **Derived-stats validation**: all six ability keys (`str/dex/con/int/wis/cha`) must be present and valid; `armor.base` and `armor.dex_cap` must be ints; `armor.shield` is treated by truthiness (missing/`false` в†’ `0`, `true` в†’ `2`) for robustness.
- **Modifier dict order** matches the spec example (`str, dex, con, int, wis, cha`).

- Invalid cases (`score=0/31`, string score, missing score, `level=0`, missing ability) all return `400`.
