### Root cause
The `bundle install` **setup step** failed before bundler even started. A prior `gem update --system` (rubygems-update-4.0.16) had written a **stale duplicate default-gemspec** for `bundler-4.0.16` into the Homebrew Cellar:

```
/opt/homebrew/Cellar/ruby/4.0.5/lib/ruby/gems/4.0.0/specifications/default/bundler-4.0.16.gemspec
```

That gemspec's `full_gem_path` resolves to `/opt/homebrew/Cellar/ruby/4.0.5/lib/ruby/gems/4.0.0/gems/bundler-4.0.16` ‚ÄĒ a directory that was **never created** (the Cellar only ships `bundler-4.0.11`; the real 4.0.16 gem files live in `/opt/homebrew/lib/ruby/gems/4.0.0/gems/bundler-4.0.16/`). RubyGems dedups default specs and this broken Cellar spec **shadowed the valid one**, so the `bundle` wrapper's `Gem.activate_and_load_bin_path(...)` computed a dangling `exe/bundle` path ‚Üí `LoadError`.

The bundler *library* itself loaded fine from site_ruby/stdlib; only the `bundle` executable wrapper was broken by the dangling path.

### Fix (one surgical symlink)
```bash
ln -sfn /opt/homebrew/lib/ruby/gems/4.0.0/gems/bundler-4.0.16 \
        /opt/homebrew/Cellar/ruby/4.0.5/lib/ruby/gems/4.0.0/gems/bundler-4.0.16
```
This makes the stale Cellar gemspec resolve to the real gem dir (which has a working `exe/bundle`). Non-destructive, minimal, and robust regardless of which default spec wins.
