## Summary

I inherited the existing D&D REST API (core + characters + combat-state suites, all passing) and added the **auth-users** stage on top, preserving all prior behavior.

### What I added to `src/server.ts`

**1. Real password hashing (Node built-in `crypto`)** ‚Äî not a toy hash:
- `hashPassword()` uses `crypto.scryptSync` with a random 16-byte salt, stored as `scrypt$<saltHex>$<hashHex>`.
- `verifyPassword()` recomputes the hash and compares with `crypto.timingSafeEqual` (constant-time, no string comparison).
- The plain password is never persisted or echoed ‚Äî register returns only `{username, role}`, login returns only `{username, token}`.

**2. `POST /v1/auth/register`** ‚Äî validation:
- `username`: 2‚Äì32 chars, `^[a-z0-9_-]{2,32}$` ‚Üí else `400`
- `password`: string ‚â• 8 chars ‚Üí else `400`
- `role`: `dm` or `player` ‚Üí else `400`
- duplicate username ‚Üí `409`
- non-object body ‚Üí `400`

**3. `POST /v1/auth/login`**:
- non-string `username`/`password` ‚Üí `400` (malformed)
- unknown user or wrong password ‚Üí `401` (bad credentials)
- success ‚Üí `{ "username": "dm", "token": "session-dm" }` (deterministic per spec)
