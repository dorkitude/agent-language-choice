## Summary

I added the **Monster and Item Compendium** APIs (Stage 5) to the existing D&D REST API, backed by SQLite, while preserving all prior-stage behavior.

### Changes to `src/storage.ts`
- **New tables** in `initSchema()`: `monsters` (slug PK, name, cr, armor_class, hit_points), `monster_tags` (slug, tag, position — FK to monsters with cascade delete), and `items` (slug PK, name, type, rarity, cost_gp).
- **New accessors**: `monsterExists`, `insertMonster`, `getMonster` (reassembles ordered tags), `itemExists`, `insertItem`, `getItem`.
- **`resetStorage()`** now drops the three new tables (children before parents for FK safety) so resets stay clean.

### Changes to `src/server.ts`
- **Handlers**: `handleCreateMonster` (201, response omits tags), `handleReadMonster` (200, response includes tags), `handleCreateItem` (201), `handleReadItem` (200).
- **Validation**: `400` for malformed input (missing/wrong-typed fields), `409` for duplicate slugs, `404` for unknown records.
- **Routing**: `POST /v1/compendium/monsters` and `POST /v1/compendium/items` added to the POST block; `GET /v1/compendium/monsters/:slug` and `GET /v1/compendium/items/:slug` added as segment-based GET routes.

### Verified
- TypeScript 7.0.2 compiles with zero errors (strict mode).
- All 4 compendium endpoints match the spec exactly (create returns 201, reads return 200, tags only on monster read).
- Full regression: dice, ability checks, encounters, initiative, characters, combat state, auth (201/409/200/401), and storage status/reset all still pass.

`./run.sh` is unchanged and ready — it compiles with the pinned `tsc` and starts the server in the foreground on `127.0.0.1:$PORT`. 🗡️
