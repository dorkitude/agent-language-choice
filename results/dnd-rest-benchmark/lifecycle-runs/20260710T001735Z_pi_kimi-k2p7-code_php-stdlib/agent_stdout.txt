Bug fixed, Chef! 🐉🔧

The failing test expected the active combatant’s key to remain in the `conditions` object even after all its conditions expired. The old code was `unset()`-ing the combatant key when the condition list became empty, so the response returned `"conditions": {}`.

**Changed in `index.php`:**  
Instead of removing the key, expired conditions now leave an empty array under the target’s name:

```php
// before:
if (count($remaining) > 0) {
    $session['conditions'][$activeName] = $remaining;
} else {
    unset($session['conditions'][$activeName]);
}

// after:
$session['conditions'][$activeName] = $remaining;
```

I verified the exact failing sequence locally: after advancing through the full order twice, the fighter’s condition expires and the response is now:

```json
{"id":"enc-1","round":2,"turn_index":2,"active":{"name":"fighter","score":14},"conditions":{"fighter":[]}}
```

All other behavior is preserved. `./run.sh` is ready and starts the PHP built-in server on `127.0.0.1:$PORT` in the foreground. 🚀
