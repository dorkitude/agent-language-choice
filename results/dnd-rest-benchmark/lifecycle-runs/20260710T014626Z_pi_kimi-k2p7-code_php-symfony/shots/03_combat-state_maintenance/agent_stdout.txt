I added the three new combat-state endpoints to the existing Symfony-routed PHP app:

- `POST /v1/combat/sessions` — creates a combat session, sorts initiative by score desc → dex desc → name asc, and returns the initial round, turn, active combatant, and order.
- `POST /v1/combat/sessions/{id}/conditions` — attaches a timed condition to a combatant.
- `POST /v1/combat/sessions/{id}/advance` — advances to the next combatant, wraps the round correctly, and decrements conditions on the newly active combatant, removing expired ones.

All previous endpoints (`/health`, dice, ability checks, XP, initiative, and character endpoints) are untouched and still working.

State is persisted across requests with a small JSON file (`combat_state.json`), and `run.sh` clears it on each fresh start before launching the PHP built-in server in the foreground on `127.0.0.1:$PORT`.

I verified the example flows and edge cases (duplicate IDs, unknown sessions, invalid targets) manually, and `./run.sh` is ready to go.
