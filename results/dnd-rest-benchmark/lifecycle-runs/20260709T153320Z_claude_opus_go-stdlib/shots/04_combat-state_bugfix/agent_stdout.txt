**Root cause:** In `conditionsMap`, combatants with zero conditions were skipped entirely (`if len(c.Conditions) == 0 { continue }`). When the fighter's `blessed` condition expired and was removed, the `"fighter"` key vanished from the response — but the evaluator expects a combatant that *once had* a condition to keep its key (as an empty list `[]`).

**Fix (3 minimal edits in `main.go`):**
- Added a `HadCondition bool` field to the `combatant` struct 🏷️
- Set `target.HadCondition = true` when a condition is added
- Changed the skip guard to `if len(c.Conditions) == 0 && !c.HadCondition` so ever-conditioned combatants retain their key

**Verified behavior:**
```
adv2 (fighter's turn, blessed decrements to 0): {"conditions":{"fighter":[]}, ...}
adv3:                                           {"conditions":{"fighter":[]}, ...}
```

The `"fighter"` key now persists as an empty array after expiry — exactly what `combat-advance-expired-removed` wants, with no regression to the other passing tests. `run.sh` is unchanged (`go run .`) and the build compiles clean. ✨
