Fixed and verified, Chef! 🎯

**Root cause:** `conditionsView` in `store.ts` only emitted a combatant's key when it currently had ≥1 condition. So the moment `fighter`'s `blessed` condition decremented to `0` and was filtered out, the `"fighter"` key vanished entirely — the evaluator's `combat-advance-expired-removed` test expected the key to remain with an empty array `[]`.

**The fix (minimal, deterministic):**
- Added an `everHadConditions: boolean` flag to the `Combatant` type
- Set it to `true` when a condition is added (`conditions/route.ts`)
- Initialize it to `false` at session creation (`sessions/route.ts`)
- `conditionsView` now includes a combatant if it has current conditions **or** ever had one

Live-tested the full failing sequence end-to-end: the final advance now returns `{"conditions":{"fighter":[]}}` instead of `{}`, and all prior behavior (decrement, wrap-round, non-active combatants not decrementing) is preserved. `./run.sh` is unchanged and ready. Server was started only for verification and is now stopped. ✨
