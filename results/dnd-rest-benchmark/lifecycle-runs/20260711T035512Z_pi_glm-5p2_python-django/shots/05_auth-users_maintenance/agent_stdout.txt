## Summary

Added the **auth-users** maintenance stage to the existing Django D&D REST API, preserving all prior endpoints (health, dice, checks, encounters, initiative, characters, combat).

### Changes

**`dndsite/views.py`** — Added two new views at the bottom:
- `POST /v1/auth/register` — validates `username` (2–32 chars, `[a-z0-9_-]`), `password` (≥8 chars), `role` (`dm`/`player`); returns `400` on malformed input, `409` on duplicate username, `{"username","role"}` on success.
- `POST /v1/auth/login` — verifies credentials; returns `400` on missing/malformed fields, `401` on bad credentials, `{"username","token":"session-<username>"}` on success.

**`dndsite/urls.py`** — Added the two new routes.

### Key design decisions
- **Password hashing**: Uses Django's built-in `make_password`/`check_password` (PBKDF2-SHA256). Works standalone with `INSTALLED_APPS = []` — no DB/migrations needed. Plain passwords are never stored or echoed.
- **In-memory user store** (`_USERS` dict + `_USER_LOCK`), following the existing combat-session pattern.
- **Status codes**: `400` (malformed), `401` (bad credentials), `409` (duplicate) — matching the spec exactly.
- All prior-stage behavior preserved (verified via test client).

`./run.sh` is unchanged and ready — starts `runserver` in the foreground on `127.0.0.1:$PORT`. 🗡️
