I preserved every existing endpoint and added the three stateful combat endpoints in `Main.java`:

- `POST /v1/combat/sessions` — creates a client-IDed session, computes initiative (`roll + dex`), sorts by score → dex → name, and returns `round`, `turn_index`, `active`, and `order`.
- `POST /v1/combat/sessions/{id}/conditions` — attaches a condition to a named combatant with positive `duration_rounds`.
- `POST /v1/combat/sessions/{id}/advance` — advances `turn_index`, wraps rounds, and decrements/removes conditions on the newly active combatant only.

All state is in-memory. Existing endpoints remain unchanged. `./run.sh` is still in place and starts the server in the foreground on `127.0.0.1:$PORT`.

I verified the exact spec examples and edge cases (wrap-round, condition expiry, unknown session 404, duplicate ID 400, invalid duration 400) compile and run cleanly with OpenJDK 26.0.1.
