The fix works. When fighter's condition expires at their turn, the `"fighter"` key is now preserved with an empty array (`"conditions":{"fighter":[]}`) instead of being dropped — exactly what the evaluator's `combat-advance-expired-removed` test expects.

## Summary 🐛✨

**Root cause:** In `advanceTurn`, the conditions map was built with `if (!c.conditions.isEmpty())`, so once a combatant's condition decremented to `0` and was removed, that combatant's key vanished from the `conditions` object entirely. The evaluator expected `"fighter": []` to remain.

**Fix** (three small edits in `Main.java`):
- Added a `boolean everHadCondition` flag to `Combatant`.
- Set it to `true` in `addCondition` when a condition is attached.
- Changed the advance-response loop to include any combatant where `everHadCondition` is true, emitting an empty array once their conditions expire.

`run.sh` is unchanged and ready — it compiles `Main.java` and starts the server in the foreground on `127.0.0.1` using `$PORT`. All prior behavior is preserved (conditions still only appear for combatants that have had one, so untouched combatants like rogue/mage never pollute the map), Chef. 🎲
