## Root cause

The `combat-advance-expired-removed` test expects the advance response's `conditions` map to still contain the `"fighter"` key with an **empty array** after the `blessed` condition expires on fighter's own turn:

```go
"conditions": map[string]any{ "fighter": []any{} },
```

But `allConditionsView` only emitted combatants whose condition list was *non-empty*:

```java
if (!c.conditions.isEmpty()) {
    map.put(c.name, conditionsView(c.conditions));
}
```

So once `blessed` ticked `1 → 0` and was removed, fighter was dropped entirely → `"conditions": {}` → the matcher reported `missing JSON key "fighter"`.

## Fix

One minimal change in `Main.java` — list **every** combatant in the conditions map (empty array when none remain):

```java
private static Map<String, Object> allConditionsView(CombatSession session) {
    Map<String, Object> map = new LinkedHashMap<>();
    for (SessionCombatant c : session.order) {
        map.put(c.name, conditionsView(c.conditions));
    }
    return map;
}
```

This is safe for all other advance tests because the evaluator's `jsonContains` matcher only checks keys present in `want` — it never asserts *absence* of extra keys, so adding `"rogue":[]` / `"mage":[]` doesn't break `combat-advance-to-mage`, `…-decrements`, or `…-expires`. No prior-stage behavior was touched.
