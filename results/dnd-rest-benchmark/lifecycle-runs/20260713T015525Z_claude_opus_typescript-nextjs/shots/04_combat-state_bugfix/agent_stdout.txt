Fixed and verified — 20/20 tests pass. 🎉

**Root cause** 🐛: The `conditionsView` in `app/v1/combat/store.ts` only emitted a combatant's key when its `conditions` array was non-empty. When `fighter`'s `blessed` condition decremented to `0` and got removed, the fighter key vanished entirely (`conditions: {}`), but the evaluator expects the key to persist as an empty array (`{"fighter": []}`) once a condition has ever been attached.

**Fix** 🔧: Added a `hadCondition` flag to the `Combatant` type:
- `store.ts` — `conditionsView` now includes a combatant's key when `hadCondition` is true (even if the array is now empty).
- `conditions/route.ts` — sets `hadCondition = true` when a condition is added.
- `sessions/route.ts` — initializes `hadCondition: false` on creation.

This keeps combatants who *never* had conditions (`rogue`, `mage`) absent from the view — preserving all previously passing tests — while keeping `fighter` present after its condition expires.

`./run.sh` is unchanged and ready (starts `next dev` in the foreground on `127.0.0.1:$PORT`). No packages downgraded, all prior behavior preserved. ✨
