I added the three combat-state endpoints to the existing Symfony-routed PHP API while preserving every prior endpoint:

- `POST /v1/combat/sessions` — creates a session with initiative order (`roll + dex`, tie-break by dex desc then name asc), sets `round = 1`, `turn_index = 0`.
- `POST /v1/combat/sessions/{id}/conditions` — attaches a condition with positive `duration_rounds` to an existing combatant.
- `POST /v1/combat/sessions/{id}/advance` — advances the turn, wraps the index and increments the round, and decrements/removes conditions only on the newly active combatant.

Since PHP’s built-in server reloads the script per request, I kept state in **System V shared memory** keyed by `PORT`, with a parent-PID boot check so each fresh server run starts with clean state.

I also updated the dispatcher to pass route params to controllers and verified:
- Existing endpoints still return correct results.
- New endpoints return 400 for malformed input/duplicate sessions and 404 for unknown session IDs.
- Conditions decrement only on the active combatant and are removed at 0.
- `./run.sh` remains the foreground server launcher on `127.0.0.1:$PORT`.
